## Known Gaps
- **Electron-driven E2E**: No automated test launches the actual Electron app and exercises the IPC bridge through a real BrowserWindow + preload. Will be addressed when user flows exist (issue #2+).
- **Hot reload**: Cannot be programmatically asserted; developer must verify manually.
- **Packaging (`npm run dist`)**: Not run in CI yet. SCAFFOLD-004 verifies the `electron-builder.yml` config, but the actual installer build is manual.
- **Custom titlebar (#50)**: Unit tests cover the React component + IPC contract, but the *frameless* behaviour itself (drag region, Aero Snap, double-click-to-maximize, Alt+Space, macOS traffic-light alignment) can only be verified manually via `npm run dev` — no automated coverage of the actual `BrowserWindow` chrome.
- **Paperplane rebrand + userData migration (#GH-51)**: 9 migrate-userdata unit tests cover copy/idempotent/skip/marker/log/failed-mkdir paths against a temp dir. The end-to-end "old e-frank install upgraded to new Paperplane build retains projects + secrets" flow needs a manual installer-vs-installer smoke test against a Windows nsis upgrade — `safeStorage` ciphertexts in particular should be round-tripped on the upgrade path because the Windows DPAPI-bound key is account-scoped, not path-scoped.
- **Sticky phase headers (#GH-52 #8)**: jsdom doesn't compute layout, so `position: sticky` behavior under scroll can only be eyeballed in `npm run dev` (load a long Implementation phase, scroll past its header, verify the header pins). The CSS rule itself is in `ExecutionLog.module.css` and is exercised whenever a row renders — no automated coverage of the visual scroll behavior.
- **Inline approve/reject in widget (#GH-52 #3)**: unit tests cover button presence + IPC dispatch, but the live "actively-running run hits a checkpoint, widget surfaces buttons, click sends approval, run resumes" flow is currently only exercised manually. Pair with #50's "frameless titlebar can only be eyeballed" — both stack on the same gap (no Electron-driven E2E yet).
- **Global notification toast stack (#GH-59)**: unit coverage is comprehensive for store / single-toast / stack / dispatcher contract, but several outcomes are visual-only and only verifiable via `npm run dev`: (a) the bottom-right anchor stacks correctly above all routes (jsdom doesn't compute `position: fixed` + z-index against the real AppShell layout), (b) entry slide-and-fade + exit transitions read smoothly + `prefers-reduced-motion: reduce` flattens to opacity (jsdom doesn't compute CSS animations or the reduced-motion media query in tests), (c) approval-toast pulsing accent border (animation-driven), (d) pointer-events pass-through gaps between toasts so clicks reach underlying buttons, and (e) the cross-route live flow ("start run from Projects list, navigate to Connections, wait for approval marker → toast appears bottom-right; click Approve → toast dismisses and run resumes"). Stack the manual smoke against the existing #GH-52 inline-approve smoke and the #GH-57 sticky-head smoke — same `npm run dev` session can exercise all three.
- **GH-38 Skill management — discover + install + list**: 63 unit tests cover the three new backend modules + state hook + view + dialog. Behaviors that can only be verified manually with `npm run dev`: (a) real-filesystem scan of `~/.claude/skills/` (the InMemoryFs seam covers the parser + dedupe but not actual readdir + readFile against the user's `%USERPROFILE%\.claude\skills` tree on Windows / `~/.claude/skills` on macOS), (b) `npx skills add <ref> -g -y` actually shelling out and returning correct stdout/stderr (CI environment doesn't have npm-published `skills` CLI guaranteed), (c) `claude --dangerously-skip-permissions -p '/find-skills <query>'` streaming real Claude output into the FindSkillDialog and the heuristic `^[-*•]\s+name:` regex extracting candidates from whatever format `/find-skills` actually emits (output format isn't our contract — if `/find-skills` switches to JSON-only or unstructured prose, the install cards stop appearing and the user falls back to manual install input — that fallback IS unit-tested but the structured detection path needs eyeball), (d) `shell.openPath` from the row Open button actually launching the OS file manager at the skill folder (jsdom doesn't shell out), (e) FindSkillDialog stream auto-scroll keeping the latest line visible during a long find. Cancel-while-finding + cross-process restart (start two finds back-to-back, second waits for first's exit) are unit-tested via FakeSpawner state transitions, but the actual `SIGTERM` + 5-min timeout behavior on real `claude` processes is only verifiable manually. Sidebar "Skills" nav item appears between Connections and Settings — unit tests assert NAV_ITEMS order indirectly via testid presence but the visual nav-rail layout (icon + label spacing, hover state) needs eyeball.
- **GH-57 polish (panel/log visual tweaks)**: 5 of the 8 items are visual-only and need manual `npm run dev` verification — (#1) panel left/right edges align with ticket-list table edges; (#3) progress-bar sliding-gradient overlay reads as energy-flow while running, flattens on terminal; (#4b) accent bar 4px pill + softened halo + mask-faded ends on done/failed/cancelled; (#4c) sticky head solid bg + bottom-border boundary cue + extends past the row's left padding up to the accent bar; (#5) Refresh icon rotates around its own center with no horizontal/vertical drift. jsdom doesn't compute layout so `position: sticky`, `mask-image`, `scrollbar-gutter`, and CSS animations all need eyeball. The behavioral hooks (`data-running`, `data-visible`, testid renames, `scrollTo` invocation) are unit-tested; the visual outcome is not.
